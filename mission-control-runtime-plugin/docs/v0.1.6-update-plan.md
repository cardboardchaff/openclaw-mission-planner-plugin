# Mission Control Runtime v0.1.6 update plan

Date: 2026-04-08

## 1) Origin resources reorganized

Original upstream artifacts are now preserved under:

- `mission-control-runtime-plugin/resources/originating/INSTALL.txt`
- `mission-control-runtime-plugin/**` (plugin scaffold with runtime source, skills, tests, workspace docs)
- `mission-control-runtime-plugin/resources/releases/README.md` (local build instructions for release archives; binary `.tgz` files are intentionally untracked)

This keeps the repository root clean and makes the source-of-truth inputs explicit.

## 2) Current plugin review (v0.1.5)

### Plugin surface

The plugin currently registers:

- CLI namespace: `openclaw tickets ...`
- tools: `ticket_classify`, `ticket_open`, `ticket_show`, `ticket_list`, `ticket_update`, `ticket_close`, `ticket_audit`, `ticket_bootstrap_workspace`
- hooks: `before_prompt_build`, `before_tool_call`, `after_tool_call`

### Governance model

Governed activity is enforced for:

- explicit tools: `cron`, `sessions_spawn`
- `exec` calls when background/elevated/PTY/long-running/dangerous-command patterns are detected

A bounded classification payload is required by default before governed work can proceed.

### Ticket/state model

Ticket state is persisted in OpenClaw state storage under `mission-control-runtime/tickets.json`, with:

- active-status handling
- overdue checks
- delivery verification bookkeeping
- per-tool-call audit trail

## 3) Compatibility risks when targeting OpenClaw v2026.4.5

Because v0.1.5 was released with `peerDependencies.openclaw: >=2026.4.1-beta.1`, the following need verification against 2026.4.5:

1. **Plugin SDK hook contract drift**
   - verify `before_prompt_build`, `before_tool_call`, `after_tool_call` payload fields are unchanged
   - verify return contract for `{ block: true, blockReason }` remains valid

2. **Tool naming drift in core/runtime**
   - `sessions_spawn` may have aliases or replacements in newer runtimes
   - `exec` policy fields may have changed (`background`, `elevated`, `pty`, `timeout`)

3. **CLI and audit contract drift**
   - plugin shell command `openclaw tasks audit --json` output schema may have changed
   - parser currently assumes a top-level `findings` array

4. **Workspace bootstrap interaction drift**
   - ensure managed blocks for `AGENTS.md` and `HEARTBEAT.md` still align with latest OpenClaw conventions

## 4) v0.1.6 implementation plan

### A. Release metadata

- bump plugin semver to `0.1.6`
- set peer range to explicitly include `2026.4.5` (for example `>=2026.4.5 <2027.0.0`)
- refresh install/readme snippets to 0.1.6 artifact name

### B. OpenClaw 2026.4.5 compatibility layer

- add a tool-alias map in policy evaluation so governed checks cover both old and new runtime tool identifiers
- harden `toolRequiresTicket(...)` to accept alternate parameter names when evaluating exec risk signals
- make heartbeat audit parser resilient to either legacy `findings` or versioned envelopes (for example `audit.findings`)

### C. Hook robustness and observability

- guard event payload reads with safe defaults
- when payloads are partial/unknown, log structured warnings and fail closed for governed operations
- add lightweight schema assertions around hook event objects in dev/test paths

### D. Regression coverage

Add tests that lock behavior for:

- governed-tool detection with alias tool names
- `exec` governance when runtime uses alternate parameter names
- overdue/unclassified block behavior in hook execution paths
- heartbeat-audit parser compatibility with multiple JSON shapes

### E. Migration and rollout

- publish `mission-control-runtime-0.1.6.tgz`
- upgrade/install in a staging OpenClaw v2026.4.5 environment
- run:
  - `openclaw tickets classify-test`
  - `openclaw tickets audit`
  - `openclaw tickets heartbeat-audit`
- validate governed-tool blocking + ticket lifecycle end-to-end

## 5) Definition of done for v0.1.6

- semantic version and docs updated
- plugin installs and loads cleanly on OpenClaw v2026.4.5
- governance hooks block/allow correctly under tested scenarios
- CLI audit pathways parse current runtime output without regressions
- regression tests pass in CI and local verification
